# `check_bi` 分型连笔算法与“原版缠论”定义的符合性分析

本文针对 `czsc/py/analyze.py` 中 `check_bi(bars: List[NewBar], **kwargs)`（`czsc/py/analyze.py:138`）把分型连成“笔”的实现逻辑，讨论其与常见“原版缠论”（缠中说禅原帖语义 + 市场上较主流的程序化口径）之间的一致点与偏离点。

> 重要前提：缠论原帖对“笔”的可程序化细则并非完全形式化（尤其在包含处理、端点选择、笔完成/延伸确认上存在多种实现口径）。因此，这里的“符合/不符合”只能针对“常见严格口径”逐条核对，而不是宣称存在唯一“原版算法”。

---

## 代码位置与依赖关系

- 去包含：`remove_include`：`czsc/py/analyze.py:19`
- 分型：`check_fx` / `check_fxs`：`czsc/py/analyze.py:80` / `czsc/py/analyze.py:110`
- 成笔：`check_bi`：`czsc/py/analyze.py:138`
- 在线更新与“回滚/延伸”：`CZSC.__update_bi`：`czsc/py/analyze.py:211`
- 最小笔长度：`envs.get_min_bi_len`：`czsc/envs.py:37`（默认 `6`，注释注明“6 新笔 / 7 老笔”）

## 该实现到底在做什么（规则提炼）

### 1) 输入数据口径

`check_bi` 的输入 `bars` 必须是“无包含关系K线”（`NewBar` 序列）。在 `CZSC.update` 中，通过 `remove_include(k1, k2, bar)`（`czsc/py/analyze.py:279-284`）把原始K线合成 `bars_ubi`，再在 `bars_ubi` 上找分型、找笔。

这属于缠论里常见的“先合并包含，再谈分型/笔”的程序化路线。

### 2) 分型定义

`check_fx` 使用典型 3K 分型：

- 顶分型：中间K `high`、`low` 同时高于左右K（严格不等）：`k1.high < k2.high > k3.high` 且 `k1.low < k2.low > k3.low`（`czsc/py/analyze.py:99-101`）
- 底分型：中间K `high`、`low` 同时低于左右K（严格不等）：`czsc/py/analyze.py:103-105`

并把分型点 `fx` 设为顶的 `high` / 底的 `low`。

### 3) 生成分型序列

`check_fxs` 逐根扫描 3K，命中分型就 append（`czsc/py/analyze.py:126-135`）。

作者意图是“顶底交替”，但当前代码的“强制交替”判断条件是：

```python
if len(fxs) >= 2 and fx.mark == fxs[-1].mark:
    ...
else:
    fxs.append(fx)
```

这意味着：当 `len(fxs) == 1` 时，即使新分型与 `fxs[-1]` 同类，也会被 append，**会允许连续两个同类分型**（与注释意图相悖）。这点会影响后续“分型序列假设顶底交替”的模块（例如 `create_fake_bis` 在 `czsc/py/objects.py:162` 里直接 `assert fx1.mark != fx2.mark`）。

### 4) 成笔端点选择与成笔条件

`check_bi` 做的是：在当前 `bars` 中**尝试找“第一笔”**（找到就返回 `BI` + 剩余未成笔K线；找不到就返回 `None` + 原 bars）。

关键点：

1. 取 `fx_a = fxs[0]` 作为起点分型（`czsc/py/analyze.py:149`）
2. 若起点是底分型，则向上笔：从后续分型里找所有“顶分型且发生在之后且 `x.fx > fx_a.fx`”，取其中 `high` 最大者作为 `fx_b`（`czsc/py/analyze.py:150-154`）
3. 若起点是顶分型，则向下笔：找后续“底分型且 `x.fx < fx_a.fx`”，取其中 `low` 最小者作为 `fx_b`（`czsc/py/analyze.py:155-158`）
4. 用 `fx_a.elements[0].dt` 到 `fx_b.elements[2].dt`（注意是终点分型的第3根K）截取笔的 bars（`czsc/py/analyze.py:166`），这等价于“终点分型必须被其右侧K确认”
5. 成笔附加条件：
   - `fx_a` 与 `fx_b` 的价格区间**不能互相包含**（`ab_include` 为 False，`czsc/py/analyze.py:170-173`）
   - 笔长度 `len(bars_a)` >= `min_bi_len`（`czsc/py/analyze.py:144-175`；默认 `6`，见 `czsc/envs.py:37-40`）

### 5) “笔延伸/回滚”的处理方式（很关键）

`CZSC.__update_bi` 在 append 新笔之后，会做一次“后处理”：如果最新的 completed 笔在随后 `bars_ubi` 里被更高/更低的价格突破，则把这笔弹出、把其 bars 与 bars_ubi 合并回去（`czsc/py/analyze.py:245-253`），以便重新计算端点。

这在工程效果上相当于：

- 先按当前已确认分型生成一笔
- 若后续走势打破该笔的极值（但可能尚未形成新的相反分型），则认为上一笔端点不成立，回滚重算

它对应了缠论语义里“笔并非一形成就永远固定，可能随着后续走势延伸/修正”的那部分，但实现上是“丢弃重算”，而不是“就地延伸更新端点”。

## 与“原版缠论”常见口径的对照

下面按缠论里最常被程序化的几个要点逐条核对。

### 一致点（符合度较高）

1) **先去包含再找分型/笔**  
这是程序化缠论里非常主流的处理路线；`CZSC.update -> remove_include -> bars_ubi` 与此一致。

2) **分型使用 3K 定义且要求确认**  
`check_bi` 截取 bars 时取到 `fx_b.elements[2].dt`（终点分型的第3根K），等价于终点分型必须被右侧确认，符合“分型需确认”的常见口径。

3) **笔长度有最小根数约束（新笔/老笔）**  
`min_bi_len` 可配置，默认 6；注释明确“6 新笔 / 7 老笔”（`czsc/envs.py:37-40`）。这与市场上对“老笔/新笔根数”的常见实现一致。

4) **允许笔端点在未出现相反分型前被修正**  
通过 `__update_bi` 的回滚机制实现“端点不稳则撤销重算”，与缠论“笔的延伸/修正”语义是相容的（尽管实现方式偏工程化）。

### 偏离点 / 争议点（不等价或依赖特定口径）

1) `check_fxs` 的“顶底交替”实现存在明显瑕疵  
如上所述（`czsc/py/analyze.py:131` 的 `len(fxs) >= 2`），会允许 `fxs` 出现连续同类分型。这与“分型序列应交替”这一强假设不一致，也会影响后续以“交替”为前提的内部分析。

2) `fx_b` 的筛选条件过弱，更多依赖“取极值”兜底  
例如向上笔里 `x.fx > fx_a.fx`（顶的 high > 底的 low）在绝大多数情况下天然成立，因此真实的筛选几乎由 `max(high)` 决定（`czsc/py/analyze.py:152-154`）。  
“原版缠论”的常见程序化口径里，端点选择往往更强调“相邻可成立分型 + 延伸更新”，而不是一次性在当前窗口里挑“所有后续顶分型中的最高点”作为终点。

3) “端点分型区间不包含”(`ab_include`) 是一种实现口径，不是所有人都认为是原帖硬条件  
`check_bi` 把“顶底分型之间没有包含关系”作为成笔必要条件（`czsc/py/analyze.py:169-173`）。  
在不少缠论实现中，这条会被弱化、改写或通过更细的包含处理来消解；因此它更像“工程化防抖规则”，是否属于“原版定义”取决于你采用的解释体系。

4) 去包含的方向判定与合并细节并非原帖唯一口径  
`remove_include` 的方向用 `k1.high` 与 `k2.high` 的关系决定（`czsc/py/analyze.py:38-41`），并据此用 max/min 组合高低点（`czsc/py/analyze.py:50-58`）。  
缠论原帖关于包含处理的叙述较抽象；在程序化时常会引入“趋势方向”“前一段走势”等信息决定合并方向。这里的简化可能在复杂震荡里导致合并结果与另一套“原版口径”不同，进而导致分型/笔差异。

5) “笔已成”与“笔未完成”的界线被工程化为“出现确认分型就算成笔”  
该项目里，一旦满足 `min_bi_len` 且终点分型已确认，就返回一笔；随后若走势继续创新高/新低再回滚重算（`czsc/py/analyze.py:245-253`）。  
这在效果上可用，但与一些缠论读者的理解（例如更强调“笔完成后再等待对向确认”等）未必等价。

## 结论（回答“是否符合原版缠论定义”）

- 如果你把“原版缠论的可程序化部分”理解为：**去包含K线 -> 3K分型 -> 端点分型确认 -> 满足新笔/老笔根数 -> 允许端点延伸修正**，那么 `check_bi` + `__update_bi` 的组合实现与该口径**总体一致**。
- 但它并不等价于任何“唯一的原版算法”，且存在若干明显的工程化取舍/瑕疵（尤其是 `check_fxs` 交替判断的条件错误、`fx_b` 的筛选偏“取极值”、以及 `ab_include` 作为硬条件这一口径争议）。因此更准确的表述是：  
  **它是“缠论笔识别的一种可用实现”，更贴近常见的新笔程序化口径，但不能宣称严格等同于原帖语义下的唯一‘原版定义’。**

---

## 如果你希望更“贴近原版/更稳定”，优先关注的改进点

1) 修正 `check_fxs` 的交替约束条件  
把 `len(fxs) >= 2` 改为 `len(fxs) >= 1` 才能真正阻止连续同类分型进入序列（并建议明确处理“同类分型取更极值替换”还是“直接跳过”）。

2) 明确你采用的“端点选择口径”  
如果你希望“相邻可成立分型 + 逐步延伸”的风格，可以把 `fx_b` 的选取从“所有后续分型取极值”改为“最近满足条件者先成笔，再在同向延伸阶段更新端点”。

3) 重新评估 `ab_include` 是否应作为硬门槛  
在某些震荡结构里，强行禁止端点区间包含会显著减少可成笔数量；是否保留，取决于你对“原版”解释与实盘噪声容忍度。
