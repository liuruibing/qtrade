# `CZSC.__update_bi` 的“笔延伸/回滚”机制：过程细化与长笔成因分析

本文专门解释 `czsc/py/analyze.py` 中 `CZSC.__update_bi` 的“笔延伸/回滚”（作者注释称“破坏”）后处理逻辑，并分析为什么它有时会画出**很长的一笔**，而在相同 K 线下其它缠论库可能会拆成多笔。

涉及代码位置：

- 成笔：`check_bi`：`czsc/py/analyze.py:138`
- 更新笔 & 回滚：`CZSC.__update_bi`：`czsc/py/analyze.py:211`（回滚段 `czsc/py/analyze.py:245-253`）
- 无包含K线：`remove_include`：`czsc/py/analyze.py:19`（由 `CZSC.update` 调用）

---

## 1. 关键数据结构：为什么 `bars_ubi` 会“回退”

这个库并不是在原始K线上直接找笔，而是维护两条序列：

1) `bars_raw`: 原始K线（`RawBar`）  
2) `bars_ubi`: 去包含后的K线（`NewBar`，无包含K线序列，作者命名为“未完成笔的无包含K线”）

每次 `CZSC.update(bar)`：

- 先把原始 `RawBar` 追加/更新进 `bars_raw`
- 再用 `remove_include` 逐根把 `bars_raw` 压缩成 `bars_ubi`（`czsc/py/analyze.py:271-285`）
- 最后调用 `__update_bi()` 尝试从 `bars_ubi` 中“确认”新的笔，或撤销最后一笔并回退重算

因此，你在图上看到的“长笔”，本质来自于：

- `check_bi` 的“端点选择策略”（偏向一次性找同向的极值端点）
- 以及 `__update_bi` 的“最后一笔可撤销并回退重算”机制（让最后一笔在趋势延伸时不断变长）

---

## 2. `check_bi` 产出一笔时，`bars_ubi` 会被截断到哪里？

当 `check_bi(bars)` 找到一笔 `(bi, bars_b)` 时（`czsc/py/analyze.py:166-176`）：

- 这笔使用的 bars 区间是：
  - 从 `fx_a.elements[0].dt`（起点分型左K）
  - 到 `fx_b.elements[2].dt`（终点分型右K，用于“分型确认”）

- **更关键的是**：它返回的“剩余 bars”（即 `bars_b`）不是从 `fx_b.dt` 或 `fx_b.elements[2].dt` 开始，而是：

```python
bars_b = [x for x in bars if x.dt >= fx_b.elements[0].dt]
```

也就是说：当一笔完成后，`bars_ubi` 会被截到“终点分型的左K”处重新开始保留（终点分型3根K的第一根），以便后续分型识别能继续覆盖“分型三根K”的窗口。

这会导致一个现象：

- 新一笔的寻找窗口，总是与上一笔的尾部**存在重叠**（至少重叠 2 根无包含K线的时间段）
- 这种重叠对于“分型确认”是合理的，但也意味着后续出现突破时，库有足够信息把上一笔撤销并重算

---

## 3. “笔延伸/回滚”详细过程（逐步描述）

以下描述针对 `__update_bi` 在正常（已有若干笔）情况下的处理流程：

### 3.1 先尝试“确认一笔”

每次 `__update_bi` 被调用时（`czsc/py/analyze.py:237-244`）：

1) 用当前 `bars_ubi` 调用 `check_bi(bars_ubi)`  
2) 若返回了 `bi`，就把它 append 到 `self.bi_list`  
3) 不管是否成功成笔，都会把 `self.bars_ubi` 更新为 `check_bi` 返回的 `bars_ubi_`

这一段的关键含义是：

- 该库是“边走边确认笔”的：只要窗口里满足成笔条件，它就会立即生成一笔并截断 `bars_ubi`
- 但“确认”并不代表永远不变，因为紧接着还有“破坏检查”

### 3.2 再做“破坏检查”：一旦突破上一笔极值，就撤销最后一笔

`__update_bi` 的后处理（`czsc/py/analyze.py:245-253`）逻辑是：

```python
last_bi = self.bi_list[-1]
bars_ubi = self.bars_ubi
if (last_bi.direction == Up and bars_ubi[-1].high > last_bi.high) \
    or (last_bi.direction == Down and bars_ubi[-1].low < last_bi.low):
    self.bars_ubi = last_bi.bars[:-2] + [x for x in bars_ubi if x.dt >= last_bi.bars[-2].dt]
    self.bi_list.pop(-1)
```

把它翻译成人话，就是：

- 如果最后一笔是向上笔（Up），但**后来出现的最新无包含K线**（`bars_ubi[-1]`）最高价超过了这笔的最高价 `last_bi.high`，那么认为“这笔的终点被破坏”，需要撤销。
- 如果最后一笔是向下笔（Down），但最新无包含K线最低价跌破了这笔的最低价 `last_bi.low`，同理撤销。

撤销时做两件事：

1) `pop` 掉最后一笔（`self.bi_list.pop(-1)`）
2) 把“撤销的那笔的 bars”和“当前 bars_ubi”合并回 `self.bars_ubi`，以便下一轮重新计算：
   - 取 `last_bi.bars[:-2]`：把被撤销的笔的大部分 bars 放回（但丢掉尾部 2 根）
   - 再拼接 `bars_ubi` 中从 `last_bi.bars[-2].dt` 开始的部分

这里的 `[:-2]` 是一个**非常工程化**的处理：作者注释写得很直白——“最后一根无包含K线有可能是未完成的，必须是 -2”（`czsc/py/analyze.py:250-252`）。这说明该回滚逻辑本身也承认“多一根K线就可能导致错误”。

### 3.3 回滚不会在当前 tick 立刻重算，而是“留到下一次 update”

注意：回滚发生在 `__update_bi` 的末尾，并不会在同一个 `update()` 里再次调用 `check_bi` 重算。

因此你在图上看到的效果往往是：

- 某一时刻先画出一笔
- 后面某根K线突破后，这笔突然消失（撤销）
- 再过若干根K线后，出现一根“更长”的同向笔（用更新后的 `bars_ubi` 重算出来）

## 4. 为什么这个库容易得到“很长的笔”（对比其它库）

导致“长笔”的首要原因，**不在回滚本身**，而在 `check_bi` 的端点选取策略。

### 4.1 `check_bi` 不是“找最近能成笔的端点”，而是“在候选里直接取极值端点”

对于起点 `fx_a`（总取 `fxs[0]`，`czsc/py/analyze.py:149`），它找终点 `fx_b` 的方式是：

- 若 `fx_a` 为底分型（做向上笔）：
  - 在**所有后续顶分型**里筛一遍，然后取 `high` 最大的那个作为 `fx_b`（`czsc/py/analyze.py:152-154`）
- 若 `fx_a` 为顶分型（做向下笔）：
  - 在**所有后续底分型**里筛一遍，然后取 `low` 最小的那个作为 `fx_b`（`czsc/py/analyze.py:157-158`）

这意味着：只要当前 `bars_ubi` 窗口足够长，`check_bi` 会倾向于把“起点之后出现过的所有同向极值端点”直接当成这一笔的终点。

而许多其它缠论库的思路更接近“在线构造”：

- 先找最近满足条件的相反分型作为“暂定端点”
- 在同向继续延伸但未出现有效反向笔前，才更新端点
- 一旦反向笔确认，前一笔结束，后续再单独形成下一笔

两者差别在于：**这个库允许在一开始就“跳过”很多中间结构，把一段走势压缩成更少的笔**。

这会直接造成你的观察：同样的K线，别的库拆成多笔，这个库经常是一笔走到更后面的极值。

### 4.2 回滚机制会进一步“拉长最后一笔”

当最后一笔生成后，如果走势同向继续创新高/新低：

1) 该库会判定“最后一笔被破坏”（因为突破了最后一笔的极值）
2) 于是撤销这笔并把其 bars 合并回 `bars_ubi`
3) 下一轮 `check_bi` 会在更长的 `bars_ubi` 上重新选端点
4) 由于 `check_bi` 的端点策略是“取极值”，重算出来的同向笔往往会变得更长

这本质上是一种“通过撤销重算实现端点延伸”的方式。

### 4.3 这是否“更符合原版缠论”？

需要拆成两个层面讨论：

**(A) “突破极值后撤销最后一笔端点”这件事本身**  
从“笔的端点应是当下走势的有效转折点”的语义出发：如果向上笔的终点被后续价格突破，说明“向上的走势并未结束”，此前的终点不应再作为最终端点。  
因此，“突破就撤销最后一笔并允许延伸”这个大方向是合理的。

**(B) 这个库的实现是否是‘正确的延伸方式’？**  
争议主要在两点：

1) 它通过“撤销整笔 + 回退 bars 重算”来延伸端点，属于工程上可行，但非常依赖细节（作者也提示“容易出错”）。  
2) 更关键：它延伸后的端点，是用 `check_bi` 的“全窗口取极值”选出来的，这会把很多其它实现会拆分出来的中间笔结构吞进同一笔里。  
   - 如果你坚持“笔应连接经过规则过滤后的相邻顶底分型（在线递推）”，那么这种“取极值跳跃式端点”会被认为不够“原版/不够严格”。
   - 如果你采用更“去噪”的口径，把大量中间摆动视为未形成有效反向笔（特别是在 `min_bi_len`、`ab_include` 较严格时），那它会显得“笔更少、更长”，但未必是错，只是口径更粗。

我的判断（结合你描述的“别的库是几笔，这里是一笔很长”）：

- **回滚/撤销本身并不离谱**（属于合理的“端点失效→重算”）
- **真正导致“过长笔”的，是 `check_bi` 的端点策略偏向全局极值**；这更像“用极值简化走势”，而不是很多人理解的“按相邻分型在线生成笔”

所以，“这个库的笔延伸算法对不对”要看你对“笔”的口径：

- 若你追求与多数缠论实现一致的“在线、相邻分型递推”的笔序列：这个实现**偏离较大**，容易把中间可成笔的反向段吞掉，从而“笔更少更长”。
- 若你更在意“过滤噪声、抓更大波段”的笔：它会显得“更稳定更粗”，但这属于**另一种工程口径**，不宜称为唯一“原版标准”。

## 5. 你可以如何验证“长笔是否来自端点取极值”

一个很直接的验证思路（不改代码也能看出）：

1) 找到图上那根“很长的笔”的起点分型 `fx_a`
2) 在它与终点之间，检查是否存在满足 `min_bi_len` 的“反向分型对”（其它库可能把它们连成了中间几笔）
3) 再对照本库 `check_bi` 的硬条件：
   - 端点区间是否被 `ab_include` 拒绝（`czsc/py/analyze.py:170-173`）
   - 反向段是否因为无包含K线数量不足而无法成笔（`min_bi_len`）

若中间确实有满足这些条件的结构，但本库仍然直接连成一笔，那么基本可以确定：**不是门槛导致，而是端点选择“取极值跳过”导致**。

---

## 6. 如果你想让它更像“多笔拆分”的口径，改动方向是什么

只给方向，不在本文直接改代码：

1) 把 `check_bi` 的 `fx_b` 选取从“全窗口取极值”改为“按时间顺序找第一个可成笔端点”，然后在同向延伸时再更新端点。  
2) 明确“同类分型如何处理”（例如同类取更极值替换，而不是简单 append），否则分型序列噪声会传导到笔。  
3) 回滚逻辑最好不要依赖 `bars[:-2]` 这种经验截断，而是显式基于“最后一笔终点分型的三根K”来构造回退窗口，减少“多一根K线就错”的脆弱性。
